## 3. Новые замечания — что делать дальше

### 3.1 КРИТИЧНО: Google Fonts загружаются с внешнего CDN

**Проблема:**
```html
<link rel="stylesheet" href="https://fonts.googleapis.com/css2?family=Inter:wght@300;400;500;600;700&family=Space+Grotesk:wght@400;500;600;700&display=swap">
```

Это создаёт два риска:

1. **Скорость в РФ.** fonts.googleapis.com и fonts.gstatic.com иногда медленно отвечают российским пользователям — это съедает Core Web Vitals (LCP растёт на 200–800 мс). Для B2G-сайта, ориентированного на госзаказчиков, это критично.

2. **Потенциальная блокировка.** Хотя сам fonts.googleapis.com пока работает в РФ, он принадлежит Google и теоретически может быть ограничен. Для сайта с лицензиями ФСТЭК/ФСБ/МО это репутационный риск — зачем зависеть от Google?

**Решение: self-hosted шрифты.**

Шаги:

1. Скачать Inter и Space Grotesk с https://fonts.google.com или https://gwfh.mranftl.com/fonts (удобный сервис: «Google Web Fonts Helper» — сразу даёт CSS и .woff2)
2. Положить .woff2 файлы в `public/fonts/`
3. Создать `public/fonts/fonts.css`:

```css
@font-face {
  font-family: 'Inter';
  font-style: normal;
  font-weight: 300 700;
  font-display: swap;
  src: url('/fonts/inter-cyrillic-latin.woff2') format('woff2');
  unicode-range: U+0000-00FF, U+0131, U+0152-0153, U+02BB-02BC, U+02C6, U+02DA, U+02DC, U+0300-0301, U+0303-0304, U+0308-0309, U+0323, U+0329, U+2000-206F, U+2074, U+20AC, U+2122, U+2191, U+2193, U+2212, U+2215, U+FEFF, U+FFFD, U+0400-045F, U+0490-0491, U+04B0-04B1, U+2116;
}

@font-face {
  font-family: 'Space Grotesk';
  font-style: normal;
  font-weight: 400 700;
  font-display: swap;
  src: url('/fonts/space-grotesk-cyrillic-latin.woff2') format('woff2');
}
```

4. В Astro layout заменить внешний `<link>` на:

```html
<link rel="preload" as="font" type="font/woff2" href="/fonts/inter-cyrillic-latin.woff2" crossorigin>
<link rel="stylesheet" href="/fonts/fonts.css">
```

5. Удалить preconnect для fonts.googleapis.com и fonts.gstatic.com — они больше не нужны.

**Ответственный:** Developer  
**Эффект:** LCP -200..800 мс, нулевая зависимость от Google, нулевой репутационный риск.

### 3.2 КРИТИЧНО: Внутренние ссылки без trailing slash

**Проблема:**

Все внутренние ссылки идут без `/` в конце:
```html
<a href="/products">Продукты</a>
<a href="/about">О компании</a>
<a href="/blog">Блог</a>
```

Но canonical и sitemap — с `/`:
```
https://azoneai.ru/products/
https://azoneai.ru/about/
```

В Object Storage это работает так: запрос `/products` отдаёт либо редирект 301 на `/products/`, либо 404 (зависит от настройки). В любом случае — **лишний HTTP-запрос на каждое кликанье навигации**.

Подсчёт: на главной 17 ссылок без slash, на /products — 15, на /blog — 15. Каждая такая ссылка = +50–200 мс на клик.

**Решение:** В Astro layout и компонентах навигации заменить:
```html
<a href="/products">  →  <a href="/products/">
<a href="/about">     →  <a href="/about/">
<a href="/blog">      →  <a href="/blog/">
<a href="/contact">   →  <a href="/contact/">
<a href="/search">    →  <a href="/search/">
```

Плюс все ссылки на продукты и статьи блога аналогично.

В `astro.config.mjs` убедиться:
```js
export default defineConfig({
  trailingSlash: 'always',
  build: {
    format: 'directory'
  }
});
```

**Ответственный:** Developer  
**Эффект:** Убирает 1 лишний редирект на каждую навигацию, улучшает SEO (Яндекс не любит цепочки редиректов), ускоряет клики на 50–200 мс.

### 3.3 ВЫСОКИЙ ПРИОРИТЕТ: Google Search Console

В dist вижу только файл верификации Яндекса. Если Google Search Console ещё не подключён — сделать это сегодня:

1. Открыть https://search.google.com/search-console
2. Добавить ресурс «URL-префикс»: `https://azoneai.ru/`
3. Выбрать метод верификации «HTML-файл»
4. Скачать файл `google<xxx>.html`
5. Положить в `public/` Astro-проекта
6. Пересобрать и задеплоить
7. Нажать «Подтвердить» в консоли
8. Подать sitemap: `https://azoneai.ru/sitemap-index.xml`

**Альтернатива:** верификация через DNS-запись TXT в Cloud DNS (предпочтительнее — не зависит от деплоя).

**Ответственный:** SEO  
**Эффект:** Индексация в Google, данные о запросах, CTR, позициях.

### 3.4 СРЕДНИЙ ПРИОРИТЕТ: Pagefind не интегрирован в layout

Поиск работает только на странице `/search/`, но на главной и в блоге нет поискового виджета в навбаре. Пользователи кликают «Поиск» → переходят на отдельную страницу → там вводят запрос.

**Улучшение:** встроить Pagefind UI в хедер или в блог-layout в виде мини-виджета. Документация: https://pagefind.app/docs/ui/

Это повысит engagement — люди будут искать чаще, если поле ввода видно сразу.

**Ответственный:** Developer  
**Эффект:** Увеличение глубины просмотра, снижение отказов.

## 4. Pagefind — новый актив

Pagefind — это серьёзный апгрейд, которого не было в моём плане. 810 КБ индекса (wasm + фрагменты), полнотекстовый поиск без сервера, всё работает через статику. Страница `/search/index.html` готова.

**Что рекомендую сделать:**

1. **Добавить поисковое поле в навбар** (не только ссылку на /search/). Документация: https://pagefind.app/docs/ui/
2. **Подключить на blog/index.html** — фильтрация статей по теме
3. **Исключить из индекса служебные страницы** (`/search/`, `/privacy/`, `/terms/`) через `data-pagefind-ignore` — чтобы в результатах поиска не было шума
4. **Pagefind metadata** — можно индексировать title, description, категории статей отдельно
