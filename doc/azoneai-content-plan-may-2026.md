### 📅 Неделя 2 — 12 мая (понедельник)

**Статья № 14. RAG-архитектура для корпоративных данных: как подключить LLM к внутренним документам**

**slug:** `14-rag-arhitektura-korporativnyh-dannyh`  
**Категория:** Технологии  
**Длина:** 10–12 минут чтения, 2200–2700 слов  
**Целевые ключи:** «RAG архитектура», «retrieval augmented generation», «LLM корпоративные данные», «семантический поиск документы»

**Почему сейчас:** RAG — это технология, на которой построен AzoneDoc. На сайте есть статья про продукт (03), но нет образовательного материала про **саму технологию**. Запрос «RAG архитектура» в Яндексе и Google растёт +40% год к году. Это идеальный мост между академическим знанием и продуктом.

**Структура:**
- В чём проблема: LLM «не знает» данные конкретной компании
- Что такое RAG в одной картинке: запрос → embedding → поиск → контекст → ответ
- Компоненты RAG: vector store, embedding-модель, reranker, LLM
- Что выбрать в России 2026: Yandex Embeddings, sbert-large-nlu-ru, FAISS vs Qdrant vs OpenSearch
- Подводные камни: чанкинг, дедупликация, контекстное окно
- Когда RAG не нужен (fine-tuning, prompt engineering)
- Кейс AzoneDoc: как мы реализовали RAG в закрытом контуре

**Ключевые ссылки:** статьи 02 (on-premise), 03 (AzoneDoc), 07 (deploy guide), 11 (privacy), продукт AzoneDoc.

**Ожидаемый эффект:** материал для архитекторов и ИТ-директоров. Длинный хвост low-frequency запросов (sbert vs e5, qdrant vs faiss и т.д.) — каждый из них приносит точечный качественный трафик.

## Универсальный шаблон каждой статьи

Чтобы выдерживать качество и не тратить время на структуру каждый раз:

### 1. Введение (150–200 слов)
- Hook: цифра, парадокс, регулятивное событие
- Какую проблему решит читатель за 10 минут
- Кому полезно (ИТ-директор / юрист / CFO)

### 2. Основная часть (1500–2500 слов)
- 4–6 секций с подзаголовками H2
- Внутри секций — H3 для подразделов, списки, таблицы
- Минимум 2 таблицы или схемы
- Реальные цифры, примеры, ссылки на источники
- Ссылки на 2–3 свои статьи и 1–2 продукта

### 3. Выводы (150–250 слов)
- 3–5 основных тезисов списком
- Что делать дальше: 2–3 практических шага

### 4. FAQ (4–6 вопросов)
- Вопросы из реального поиска (Яндекс «вместе с этим ищут»)
- Короткие ответы по 2–4 предложения
- **Обязательно** оборачивать в FAQPage Schema.org для rich snippets

### 5. CTA-блок
- «Обсудить пилотный проект» → /contact/?product=...
- «Скачать чек-лист» → /AZONE_AI_presentation.pdf

### 6. RelatedArticles + RelatedProducts
- Использовать готовые компоненты

## SEO-чек-лист для каждой публикации

При публикации каждой статьи проверять:

- [ ] Title содержит главный ключевой запрос (≤60 символов)
- [ ] Description содержит ключ + CTA (≤160 символов)
- [ ] H1 совпадает с темой (но может отличаться от title)
- [ ] 3–5 H2-подзаголовков с long-tail ключами
- [ ] articleDate проставлен через Base.astro (после унификации schema)
- [ ] image для og:image добавлен (минимум 1200×630, .webp ≤200 КБ)
- [ ] Все изображения с alt-текстами
- [ ] FAQPage schema добавлен (4+ вопроса)
- [ ] 2–3 внутренние ссылки на статьи блога
- [ ] 1–2 внутренние ссылки на продуктовые страницы
- [ ] CTA-блок в конце ведёт на /contact/
- [ ] Tag список для visual-блока в шапке
- [ ] Slug в kebab-case, понятный для человека
